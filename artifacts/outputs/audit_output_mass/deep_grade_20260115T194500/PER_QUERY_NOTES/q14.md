## Root Cause Analysis

### Primary: Intent Misclassification

The query was classified as `extraction` intent instead of inheriting `documents` from q13.

**Evidence:**
- `intent: "extraction"` in results.jsonl
- Previous query (q13) had `intent: "documents"` with 4 sources

### Secondary: Follow-up Detection Failure

The follow-up detection in `routingPriority.service.ts` did not boost documents intent.

**Evidence:**
- Query references "abril e maio de 2024" which were mentioned in q13's context
- No DOC_CONTINUITY_INTENTS blocking triggered

### Tertiary: Decision Tree Bypass

In `decisionTree.service.ts:294`, the `extraction` intent bypasses the `wasDocContext` check:

```typescript
case 'extraction':
  return 'extraction';  // Does NOT check wasDocContext!
```

## Code Trace

1. **Intent Engine** (`kodaIntentEngineV3.service.ts`)
   - Classified as `extraction` due to "Quanto foi" pattern
   - Confidence likely low but not overridden

2. **Routing Priority** (`routingPriority.service.ts`)
   - Did not apply follow-up boost
   - `extraction` is in DOC_CONTINUITY_INTENTS but blocker only applies to help/conversation

3. **Decision Tree** (`decisionTree.service.ts:294`)
   - `extraction` case returns directly without wasDocContext check
   - Should have returned `documents` since previous turn was doc-grounded

4. **Orchestrator** (`kodaOrchestratorV3.service.ts`)
   - Received `extraction` family, no sources
   - Fell through to help template response

## Minimal Fix

**File:** `backend/src/services/core/decisionTree.service.ts`
**Location:** Line 294

```typescript
// BEFORE
case 'extraction':
  return 'extraction';

// AFTER
case 'extraction':
  // CRITICAL FIX: Block extraction when previous turn was doc-related
  if (wasDocContext && hasDocs && !hasExplicitHelpKeyword) {
    return 'documents';
  }
  return 'extraction';
```
